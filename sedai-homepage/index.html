The Origin of SED

ü§î The Skepticism Behind SED's Birth

When new Vibe Coding concepts, methodologies, tools, and agents emerged, I remained deeply skeptical. All of these approaches, regardless of their packaging, boiled down to the same fundamental question: How can we effectively inject context into AI?

Whether humans communicate with AI through text, voice, or video, it's ultimately just a prompt. This fundamental nature will never change. All the fancy tools‚ÄîContext, SpecKit, Skills, MCP‚Äîare merely different vehicles for context injection.

üéØ The Reality of Vibe Coding

"Build me a website I'll love." The AI autonomously creates elaborate specifications and impressive-looking code. But here's the problem: you don't even know exactly what you want, yet the AI produces something grand that looks beautiful but proves utterly impractical in real-world use.

üîÑ Where Everything Falls Apart

The real problem begins with the second prompt. When you request modifications, your intent starts diverging from the AI's original plan. The AI begins making errors in code and logic it previously generated. You're trapped in an endless cycle of corrections‚Äîthe "hamster wheel" problem‚Äîrunning endlessly without making real progress.

‚ö° SED's Radical Solution

Provide complete specifications to AI and demand absolute obedience. Even if the spec is wrong, follow the spec exactly.

Of course, AI does have two crucial authorities:

üõë
Authority to Halt: If the specification contains critical errors that would cause severe problems, AI has the authority to stop work.
üí¨
Authority to Request Improvements: AI can point out errors in the specification and request humans to improve them.

However, AI does not have the authority to modify specifications on its own.
This is the core essence of SED‚Äîcomplete specifications plus absolute AI obedience, with limited but crucial AI authority to protect against critical errors.

How SED Works

SED transforms the relationship between developers and AI through a systematic,
specification-driven approach that eliminates ambiguity and ensures precision.

1

Write Complete Specifications

Document everything: database schemas, function names, UI/UX details, error handling, testing requirements, code comments, user-facing text, and internationalization‚Äîleaving no room for AI interpretation.

2

AI Evaluates & Scores

Before development begins, AI evaluates your specifications and assigns a score (0-100). Development proceeds only when score ‚â• 95. AI identifies gaps and requests improvements rather than making assumptions.

3

Strict Execution

AI executes exactly what is specified‚Äîno interpretation, no improvisation, no assumptions. Even if the spec is wrong, AI follows it precisely, with authority to halt only for critical errors.

üìù What Must Be Specified

SED specifications are not limited to high-level concepts. They must include:

  • Class & Function Names: Every class, method, parameter, and return type
  • Implementation Details: Algorithms, business logic, data structures
  • Database Specs: Exact versions, schemas, indexes, foreign keys
  • Code Comments: JSDoc annotations, inline explanations, file headers
  • CSS Styling: Colors, spacing, fonts, animations, breakpoints
  • User-Facing Text: Labels, buttons, messages, tooltips, placeholders
  • Internationalization: Complete translation dictionaries for all languages
  • Testing: Tools, versions, coverage targets, test cases, CI/CD pipeline

"Complete specification = everything a developer would need to know when implementing the system manually."
